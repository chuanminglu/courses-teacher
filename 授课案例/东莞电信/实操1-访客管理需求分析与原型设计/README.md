# 实操1：访客管理需求分析与原型设计

> **🎯 演练目标**：通过AI辅助完成从需求澄清到原型设计的完整流程，体验AI如何提升软件需求工程效率

---

## 📚 演练背景

**项目名称**：数据标注园区智能化管理系统 - 访客管理模块

**业务场景**：园区需要实现访客全流程线上化管理，提升安全性与便捷性。目前访客管理存在流程不规范、信息记录不完整、安全性低等问题。

**演练时长**：约2-3小时

---

## 🎯 学习目标

通过本次演练，您将学会：

1. ✅ 如何使用AI进行需求澄清和补充
2. ✅ 如何生成符合行业标准的需求规格说明书
3. ✅ 如何将需求拆分为可实施的用户故事
4. ✅ 如何设计页面原型并生成交互式HTML
5. ✅ **重要**：如何对AI生成的内容进行人工验证和优化

---

## 📋 演练流程概览

```
原始需求
    ↓
步骤1: 需求澄清（使用AI提问）
    ↓
步骤2: 生成标准需求文档（SRS）
    ↓
步骤3: 拆分用户故事（User Stories）
    ↓
步骤4: 选择Story进行原型设计
    ↓
    ├─ 4.1 分析页面元素
    ├─ 4.2 分析交互逻辑
    └─ 4.3 生成HTML原型
    ↓
步骤5: 人工验证与优化（关键环节）
    ↓
输出：可用的原型文件
```

## 📂 演练步骤详细说明

### 步骤1：需求澄清 (30分钟)

**目标**：通过AI辅助，识别需求中的模糊点和缺失点

**操作指引**：
1. 打开提示词文件：`prompts/01-需求澄清提示词.md`
2. 复制提示词到AI工具
3. 粘贴原始需求作为输入
4. 获取澄清问题清单
5. **人工审核**：根据业务实际情况回答这些问题

**输出物**：
- `outputs/01-需求澄清问题清单.md`
- `outputs/01-需求澄清回答记录.md`

**⚠️ 注意事项**：
- AI生成的问题可能过于理论化，需要结合实际业务场景筛选
- 有些问题可能需要与业务方沟通才能回答

---

### 步骤2：生成标准需求文档 (45分钟)

**目标**：将澄清后的需求转化为符合IEEE 830标准的SRS文档

**操作指引**：
1. 打开提示词文件：`prompts/02-生成需求文档提示词.md`
2. 将原始需求+澄清结果作为输入
3. 生成SRS文档
4. **人工验证**：检查文档的完整性和准确性

**输出物**：
- `outputs/02-访客管理系统需求规格说明书.md`

**✅ 验证清单**：
- [ ] 功能需求是否完整覆盖业务场景
- [ ] 非功能需求是否包含性能、安全、可用性等维度
- [ ] 约束条件是否符合实际情况
- [ ] 术语定义是否清晰

---

### 步骤3：拆分用户故事 (40分钟)

**目标**：将需求文档拆分为可独立开发的用户故事

**操作指引**：
1. 打开提示词文件：`prompts/03-拆分用户故事提示词.md`
2. 输入步骤2生成的SRS文档
3. 生成用户故事列表
4. **人工评审**：评估故事的大小和优先级

**输出物**：
- `outputs/03-用户故事列表.md`

**📊 评估标准**：
- 每个故事是否符合INVEST原则
- 故事是否可在1-2周内完成
- 优先级划分是否合理

---

### 步骤4：原型设计 (90分钟)

选择一个用户故事进行原型设计，例如："作为访客，我希望能够线上提交来访申请"

#### 步骤4.1：分析页面元素 (20分钟)

**操作指引**：
1. 打开提示词文件：`prompts/04-分析页面元素提示词.md`
2. 输入选定的用户故事
3. 获取页面元素清单
4. **人工补充**：根据业务经验补充遗漏的元素

**输出物**：
- `outputs/04-访客申请页面元素清单.md`

---

#### 步骤4.2：分析交互逻辑 (30分钟)

**操作指引**：
1. 打开提示词文件：`prompts/05-分析交互逻辑提示词.md`
2. 输入页面元素清单
3. 生成交互流程图和逻辑说明
4. **人工优化**：调整交互流程使其更符合用户习惯

**输出物**：
- `outputs/05-访客申请交互逻辑说明.md`

---

#### 步骤4.3：生成HTML原型 (40分钟)

**操作指引**：
1. 打开提示词文件：`prompts/06-生成HTML原型提示词.md`
2. 输入页面元素和交互逻辑
3. 生成HTML原型代码
4. **人工调试**：在浏览器中测试并调整样式

**输出物**：
- `outputs/06-访客申请页面原型.html`

**🎨 优化建议**：
- 调整配色方案符合企业VI
- 优化表单布局提升易用性
- 添加必要的提示信息

---

### 步骤5：人工验证与优化 (30分钟) ⭐**关键环节**

**目标**：理解AI不能替代人工判断的重要性

**验证维度**：

#### 1. 业务逻辑验证
- [ ] 审批流程是否符合实际业务规则？
- [ ] 字段验证规则是否完整？
- [ ] 异常情况处理是否合理？

#### 2. 用户体验验证
- [ ] 页面布局是否符合用户操作习惯？
- [ ] 提示信息是否清晰易懂？
- [ ] 是否考虑了移动端适配？

#### 3. 技术可行性验证
- [ ] 与现有系统的集成方案是否可行？
- [ ] 性能指标是否合理？
- [ ] 安全措施是否充分？

#### 4. 完整性验证
- [ ] 是否遗漏了重要的边界场景？
- [ ] 错误处理是否完善？
- [ ] 数据权限控制是否考虑周全？

**优化方法**：
1. 召开小组评审会议，讨论AI生成内容的合理性
2. 与业务方确认关键流程和字段
3. 进行用户测试收集反馈
4. 迭代优化直到满足实际需求

**输出物**：
- `outputs/07-验证与优化记录.md`
- `outputs/08-最终版本文档包.zip`

## 💡 常见问题

### Q1: AI生成的需求文档太通用怎么办？
**A**: 在提示词中增加更多业务上下文和具体场景描述，并在生成后进行人工细化。

### Q2: 用户故事拆分粒度如何把握？
**A**: 参考INVEST原则，确保每个故事能在1-2个迭代周期内完成。可以让团队评审拆分结果。

### Q3: HTML原型不够美观怎么办？
**A**: AI生成的原型重在功能和交互演示，美化工作可以在后续设计阶段由UI设计师完成。

### Q4: 如何确保AI生成内容的质量？
**A**: 建立验证机制：同行评审、业务方确认、用户测试，并形成验证清单作为标准。

## 📊 质量保证与人工审核

### ⚠️ 重要原则：AI助力开发 ≠ 只写提示词

AI生成的文档和代码是**第一版草稿**，需要人工审核和优化！

### 人工审核检查清单

#### 需求文档审核
- [ ] 功能需求是否可测试？有无模糊表达？
- [ ] 非功能需求是否可量化？（如"性能好"改为"P95延迟<3秒"）
- [ ] 是否遗漏关键业务规则？
- [ ] 边界条件和异常情况是否考虑充分？

#### 用户故事审核
- [ ] Story拆分粒度是否合理？（2-5天可完成）
- [ ] 验收标准是否可验证？
- [ ] 优先级和依赖关系是否合理？
- [ ] 技术实现是否可行？

#### 交互设计审核
- [ ] 异常流程是否考虑充分？（网络失败、权限不足等）
- [ ] 错误提示是否用户友好？
- [ ] 页面跳转逻辑是否符合用户习惯？
- [ ] 移动端和PC端体验是否一致？

#### HTML原型审核
- [ ] 原型在浏览器中能否正常运行？
- [ ] 表单验证规则是否正确实现？
- [ ] 交互反馈是否清晰可见？
- [ ] 代码结构是否易于理解和修改？

### 常见AI生成内容的问题

| 问题类型 | 示例 | 改进方法 |
|---------|------|---------|
| 需求模糊 | "系统性能良好" | 改为"API响应时间P95<3秒" |
| 遗漏边界情况 | 只考虑正常流程 | 补充异常流程和极端情况 |
| 验证规则不完整 | 只验证必填 | 补充格式、长度、业务规则验证 |
| 交互逻辑过于简单 | 只有成功流程 | 补充失败、超时、权限不足等场景 |

### 团队评审建议

1. **需求评审会**（30分钟）
   - 与产品经理确认功能需求完整性
   - 与架构师确认非功能需求合理性
   - 与测试团队确认验收标准可测试性

2. **交互设计评审**（20分钟）
   - 与UI设计师确认交互流程合理性
   - 与前端开发确认技术可行性

3. **原型演示**（15分钟）
   - 演示关键功能流程
   - 收集用户反馈
   - 识别改进点
